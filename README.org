An Emacs plugin for the [[https://lean-lang.org][Lean 4]] theorem prover, working
toward feature parity with [[https://github.com/Julian/lean.nvim][lean.nvim]].

Built for *Eglot* lsp client.

#+begin_quote
⚠️ *Vibe-coded.* This project is written largely by prompting an AI coding
agent. Expect rough edges; review before relying on it.
#+end_quote

* Installation

#+begin_src elisp
(add-to-list 'load-path "/path/to/neo-lean-mode")
(require 'neo-lean-mode)
#+end_src

Open a ~.lean~ file in a Lake project, start the server with ~eglot~, then run
~neo-lean-goal~ inside a proof. The goal buffer then follows the cursor.

** Requirements

- Emacs 29.1+ (developed on 30.2).
- A Lean 4 toolchain (~lake~, ~lean~) on ~PATH~, e.g. via
  [[https://github.com/leanprover/elan][elan]].

* Features

The goal buffer shows the live proof state: the interactive tactic goals and the
term goal (the expected type at point, under an =Expected type:= heading). The
goal is interactive -- each subexpression carries the server's info, so
go-to-definition/type jumps from the subexpression under point to its source
(wired into Doom's =+lookup/= commands automatically; see [[*Doom integration][Doom integration]]).

Hover works through ElDoc: move point onto a subexpression in the goal buffer
and its type and docstring appear in the echo area (use ~eldoc-doc-buffer~ for
the full, multi-line view).  The subexpression under point is highlighted as you
move, so you can see exactly which subterm the hover refers to.

If the optional [[https://github.com/casouri/eldoc-box][eldoc-box]] package is installed, the hover is instead shown in a
childframe tooltip at point (like lean.nvim's floating window). It is enabled
automatically in graphical frames; set ~neo-lean-goal-hover-tooltip~ to nil to
keep the echo-area display.

Also: Unicode abbreviation input (\to → →, \alpha → α) and file-processing
progress bars (lines still being elaborated are marked in the left fringe; a ~|~
in the margin on terminals).

| Command                           | Binding   | Evil (Doom) | Description                                                                |
|-----------------------------------+-----------+-------------+----------------------------------------------------------------------------|
| ~neo-lean-goal~                   | ~C-c C-g~ | ~SPC m g~   | Show the interactive goal at point.                                        |
| ~neo-lean-goal-go-to-definition~  | ~RET~     | ~g d~       | In the goal buffer, go to the definition of the subexpression under point. |
| ~neo-lean-goal-go-to-declaration~ | --        | --          | As above, but the declaration.                                             |
| ~neo-lean-goal-go-to-type~        | --        | ~SPC c t~   | As above, but the subexpression's type.                                    |
| ~neo-lean-restart-file~           | ~C-c C-r~ | --          | Reopen the file on the server to rebuild and reload its imports.           |
| ~neo-lean-input-activate~         | --        | --          | Turn on the Neo-Lean Unicode input method.                                 |
| ~neo-lean-progress-clear~         | --        | --          | Remove file-processing progress bars from the buffer.                      |

The =Evil (Doom)= column lists the suggested Doom bindings.

* Doom integration

Enabled automatically when Doom is detected; no configuration required. It hooks
neo-lean into various Doom features. For example, ~+lookup/definition~ works on
the symbol under point in the goal buffer.

* Development

Run the batch test suite (pure functions, rendering, RPC wire-format, the input
method -- no Lean server needed):

#+begin_src sh
make test
#+end_src

~make compile~ byte-compiles every source file, and ~make clean~ removes the
artifacts. Override the Emacs binary with e.g. ~make EMACS=emacs-30.1 test~.
CI runs these on several Emacs versions via GitHub Actions (see
~.github/workflows/ci.yml~).

The end-to-end test in ~test/e2e.el~ talks to a real ~lake serve~ and is run
manually, not by ~make test~; see the file header for instructions.
